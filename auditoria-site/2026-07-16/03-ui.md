## O que está BOM e deve ser MANTIDO

- **Hierarquia do Overview** — MANTER. "Overview" (H1) domina, e a ordem Campaign status → Results → Activity → Channels → Replies responde as perguntas do cliente na sequência certa. (`overview-desktop-fold.png`)
- **Paridade claro/escuro** — MANTER. Mesmo layout, mesmos pesos, tokens semânticos flipando corretamente. Nenhuma regressão visual entre temas.
- **Cor de destaque escassa (Von Restorff funcionando)** — MANTER. O ciano primário aparece só em estado ativo; verde/âmbar só em semântica de status. Quando algo acende, significa algo.
- **Sistema de tokens 4/8px e escala tipográfica declarada** — MANTER. Componentes consomem via camadas (primitivo → semântico → componente).
- **Trio tipográfico com papéis claros** — MANTER. Space Grotesk (display/números), Figtree (corpo/UI), JetBrains Mono (labels técnicos). Os papéis nunca se misturam — o mono uppercase como "voz de máquina" reforça a estética terminal.
- **Gauge com label fora do arco** — MANTER. `Gauge.module.css:35` documenta a decisão; o problema do bug reportado não é do Gauge (ver UI-01).
- **Login** — MANTER. Um elemento dominante, estética terminal consistente, credenciais demo visualmente subordinadas. Igualmente bom no mobile.
- **DataTable desktop + estratégia de empilhar no mobile** — MANTER o conceito (com os ajustes de UI-06).
- **Estados hover/focus existem e são consistentes** — MANTER. Todo componente interativo do DS tem `:hover` e `:focus-visible` com outline no primário (verificado no CSS).

## Achados

### [SEV 3] UI-01 — Badge "UNDER 2%" invade o label "Deliverability" — Overview — desktop, ambos os temas
- **Elemento:** badge de benchmark do Bounce rate, card Email, seção Channels.
- **Princípio violado:** integridade do layout / Aesthetic-Usability Effect — sobreposição de texto é lida como produto quebrado; num dashboard de dados isso corrói a confiança nos números.
- **Evidência:** `bug-deliverability-closeup-desktop.png` e `bug-deliverability-medicoes.json` — badge x 500,4→599,0 sobre label x 563,0→667,0: **35,97 × 11,5px de sobreposição**. No mobile não ocorre (colunas empilham a ≤520px).
- **Causa-raiz (PORTAL, não DS):** `src/components/metrics/metrics.module.css:52-61` — `.metricValue` é flex row **sem `flex-wrap`** dentro de células `minmax(0, 1fr)` (`.metricList`, linha 35) com `min-width: 0` (`.metric`, linha 41). O Badge do DS tem `white-space: nowrap` (correto para badge) e mede 98,6px; a célula do Bounce rate a 1440px é mais estreita que "1.1%" + gap + badge. Flex sem wrap não encolhe abaixo do min-content e nada faz clip → o badge transborda para a coluna `.gauges` vizinha, exatamente na faixa vertical do label do gauge. **Gauge e Badge do DS estão corretos; a correção é no portal.**
- **Solução proposta:** em `metrics.module.css`, `.metricValue { flex-wrap: wrap; row-gap: var(--dp-space-25); }`. Opcional: `column-gap: var(--dp-space-200)` no `.channelBody`.
- **Por que resolve:** com wrap, o badge desce para a linha seguinte **dentro da própria célula** (~129px disponíveis > 98,6px do badge) em vez de vazar. Robusto para qualquer largura, incl. cards detalhados (Q3/EMEA), que usam o mesmo padrão.
- **Confiança:** alta.

### [SEV 4] UI-02 — Topbar mobile: toggle de tema inacessível e avatar "SM" por cima do CampaignSelector — todas as telas — mobile, ambos os temas
- **Elemento:** shell (TopBar): triggers dos selectors × ThemeToggle × UserMenu.
- **Princípio violado:** Fitts's Law + alvo de toque (HIG ≥44pt; WCAG 2.2 ≥24px) — alvos sobrepostos têm área efetiva zero.
- **Evidência:** `topbar-mobile.png` + medições: trigger "Select campaign" x 219,5→386,6; toggle x 296,6→318,6 (**invisível, atrás do trigger**); botão "SM" x 334,6→382,6 (**por cima do trigger**). Clique no trigger interceptado pelo botão de conta (reproduzido). `replies-emea-filter-mobile.png` mostra "SM" no meio do texto "EMEA Expansion — Email".
- **Causa-raiz (DS, com agravante no portal):**
  1. **Estouro:** `ds .../Select/Select.module.css:4-18` — `.root { display: inline-block; width: 100% }` e `.value` com nowrap (linha 55). Como flex item, o mínimo automático do Select é o **min-content** = texto completo (o ellipsis nunca ativa porque nenhum item da cadeia tem `min-width: 0` no `.root`). Hambúrguer + "Acme Co." + nome da campanha não cabem em 390px → conteúdo vaza por baixo de `.right`.
  2. **Ordem de pintura:** o `.root` do Select é `position: relative` → pinta **acima** dos não-posicionados; o ThemeToggle é botão comum (sem position) → fica atrás. O UserMenu também é `position: relative` e vem depois no DOM → pinta por cima. (*Paint order*: posicionados > não-posicionados; entre posicionados vence a ordem do DOM.)
- **Solução proposta:** (a) **DS:** `Select.module.css` → `.root { min-width: 0; max-width: 100%; }` (mantém `width: 100%` para formulários, permite encolher como flex item, ativando o ellipsis existente); (b) **Portal:** em `AppShell.module.css`, ≤640px, limitar os selectors (`max-width: 42vw`) **ou** mover ClientSelector/CampaignSelector para uma linha própria abaixo do topbar. A opção (b) é a mais segura para 390px.
- **Por que resolve:** `min-width: 0` remove o piso de min-content → o trigger trunca com "…" em vez de vazar; sem vazamento, não há sobreposição e a paint order deixa de importar.
- **Confiança:** alta.

### [SEV 3] UI-03 — Scroll horizontal no mobile (76–91px) — detalhe de campanha, Replies, Reports — mobile
- **Elemento:** o mesmo trigger do CampaignSelector do UI-02 — **não são as tabelas**.
- **Evidência:** `campaign-detail-emea-mobile.png` e `replies-emea-filter-mobile.png` têm **1398px físicos ÷ 3 = 466px CSS** (390 + 76); o único elemento que alcança a borda direita é a linha do topbar com o trigger "EMEA Expansion — Email" — os cards de conteúdo terminam em ~390px. A matemática fecha: **76px = "EMEA Expansion — Email"; 91px = "Q3 Outbound — US Market"** (nome mais longo). Replies/Reports estouram porque o CampaignSelector global fica preso na campanha visitada antes — não por causa da DataTable, que tem `overflow-x: auto` e empilha no mobile (confirmado nos screenshots).
- **Causa-raiz:** idêntica ao UI-02. **Um só fix elimina UI-02 e UI-03.**
- **Solução proposta:** a mesma do UI-02. Critério de aceite: `document.documentElement.scrollWidth === 390` em todas as 6 rotas com qualquer campanha selecionada.
- **Confiança:** alta.

### [SEV 3] UI-04 — Labels do stepper de fases colidem e ficam ilegíveis — Overview (Campaign status) — mobile
- **Elemento:** `Timeline` horizontal compact (Q3 com 8 fases, EMEA com 6).
- **Evidência:** `overview-mobile.png`: lê-se "Tech Send ICP & Cop Camp Link Opti Ongoin / nical ing Data y & aign dIn ation g…" e status "ACTIVEACTIVE"/"UPCOMING" sobrepostos. A seção mais importante do portal é ilegível no mobile.
- **Causa-raiz (DS + uso no portal):** `ds .../Timeline/Timeline.module.css:76-78` — `.horizontal .phase { flex: 1 1 0; min-width: 0 }` + `.label { overflow-wrap: anywhere }`. Com 8 fases em ~334px úteis, cada coluna tem ~42px: o `overflow-wrap: anywhere` estilhaça as palavras, e o status uppercase mono ("UPCOMING" ≈ 70px) não quebra → transborda e invade as vizinhas. O portal agrava usando `orientation="horizontal"` incondicional (`OverviewPage.tsx:93,98`).
- **Solução proposta:** **DS:** comportamento responsivo no `.horizontal` — a ≤640px virar vertical (as regras `.vertical` já existem e funcionam) ou `overflow-x: auto` com `.phase { min-width: 72px }`. **Portal (alternativa imediata):** `orientation="vertical"` quando mobile (hook `useResponsive` já existe).
- **Confiança:** alta.

### [SEV 3] UI-05 — Todos os ícones `Icon` (lucide) somem silenciosamente — todas as telas — ambos viewports e temas
- **Elemento:** todo glifo servido pelo `Icon` do DS: ícones da sidebar/drawer, engrenagem do "Settings · soon", envelope do canal Email (título do card e coluna CHANNEL), check do badge "under 2%", download do botão "PDF".
- **Evidência:** `capturas/html/replies.html` — spans de ícone da sidebar **vazios** (`<span class="_icon_x1l21_72" aria-hidden="true"></span>`); célula Channel vazia nas linhas de email (só o SVG inline do LinkedIn, que não passa pelo `Icon`, aparece); `bug-deliverability.html` — badge com texto `" under 2%"` (espaço órfão do ícone não renderizado). Visível em `campaigns-desktop.png` (chip "Email" sem glifo, "LinkedIn" com) e `reports-desktop.png` (botão "PDF" sem ícone). Os únicos ícones visíveis no produto são SVGs inline.
- **Causa-raiz (DS):** `ds .../Icon/Icon.tsx:24-25` — `if (!LucideIcon || typeof LucideIcon !== 'function') return null`. Ícones do lucide-react são componentes `forwardRef` → `typeof` retorna `'object'` → o guard devolve `null` para **todos** os nomes válidos. Falha silenciosa.
- **Impacto:** o produto perde toda a camada de iconografia — navegação mais lenta, coluna de tabela aparentemente quebrada, badge com espaço estranho.
- **Solução proposta:** trocar o guard por `if (!LucideIcon) return null`. Bump + publish do DS e reinstalar no portal (e conferir o site, que também consome Icon). Depois do fix, revisar espaço extra no JSX do badge (`ChannelCards.tsx:51`).
- **Confiança:** alta na evidência; média-alta no mecanismo exato — confirmar com teste local antes do release.

### [SEV 2] UI-06 — Cards empilhados da tabela no mobile: linha "CHANNEL" vazia e densidade baixíssima — Replies, Reports, detalhe — mobile
- **Evidência:** `replies-emea-filter-mobile.png` — cada card tem linha "CHANNEL" em branco (consequência do UI-05, mas mesmo com ícone seria uma linha inteira para um glifo); Replies passa de 8.000px e o detalhe Q3 chega a ~31.500px de altura.
- **Causa-raiz:** DS `DataTable.module.css:66-91` (padrão empilhado não permite omitir/compactar colunas) + portal `RepliesTable.tsx` (não prioriza colunas).
- **Solução proposta:** (a) curto prazo: ocultar células vazias no mobile (`.td:empty { display: none }` no DS); (b) melhor: prop `mobileHidden`/prioridade por coluna na DataTable; no Replies mobile compor card "Contato + badge + snippet + data" como o "Latest interested replies" do Overview já faz — é o modelo certo.
- **Confiança:** alta.

### [SEV 2] UI-07 — Alvos de toque abaixo do recomendado no shell e nas tabs — mobile
- **Elemento:** ThemeToggle 40×40px, tabs de Replies ~35px de altura com labels "Not now"/"Not interested" quebrando em duas linhas, links "use" do login.
- **Princípio:** HIG ≥44×44pt / M3 ≥48dp; acima do mínimo WCAG (24px) → qualidade, não conformidade.
- **Solução proposta:** no DS, elevar área clicável via padding (tabs `padding-block: var(--dp-space-100)`; toggle 44px; tablist mobile com `flex-wrap: nowrap` + `overflow-x: auto`).
- **Confiança:** alta.

### [SEV 2] UI-08 — Microtexto de 10px em elementos informativos — todas as telas
- **Elemento:** headers de tabela (`--dp-font-size-25` = **10px**), status do Timeline ("DONE/ACTIVE/UPCOMING"), label do Gauge ("DELIVERABILITY") — 10px uppercase mono.
- **Solução proposta:** subir o papel "micro label informativo" para 11-12px no DS; reservar 10px para carimbos decorativos sem informação primária.
- **Confiança:** alta nos valores; média na priorização (trade-off estético da linguagem terminal).

### [SEV 1] UI-09 — Cards de campanha com blocos internos desalinhados entre si — Campaigns — desktop
- **Evidência:** título da EMEA quebra em 3 linhas vs 2 do Q3, empurrando chips, stepper e métricas para alturas diferentes (`campaigns-desktop.png`).
- **Solução proposta:** `min-height` para 2 linhas no cabeçalho + `-webkit-line-clamp: 2`, ou alinhar rodapé com `margin-top: auto`.
- **Confiança:** alta.

### [SEV 1] UI-10 — Fallbacks de token com metade do valor do token real (inconsistência latente) — portal — código
- **Evidência:** `layout.module.css:12` e `metrics.module.css:22,58` usam `var(--dp-font-size-300, 1.25rem)` — mas `size-300` = **32px** (fallback sugere 20px); `var(--dp-font-size-200, 1.125rem)` — `size-200` = **24px** (fallback 18px). Hoje renderiza 32/24px; se a intenção era 20/18px, títulos e valores estão ~60% maiores do que o desenhado.
- **Solução proposta:** decidir o valor intencional, apontar para o token certo e remover fallbacks divergentes.
- **Confiança:** alta no fato; hipótese sobre a intenção original.

---

**Nota de correção de rota para os outros especialistas:** o inventário atribuía o scroll horizontal mobile às tabelas; a evidência mostra que **a causa é o CampaignSelector no shell** (largura das capturas = 466px; único elemento na borda direita é o topbar), valendo para qualquer página com campanha de nome longo selecionada. UI-02 e UI-03 se resolvem juntos: fix no DS (`Select`) + ajuste de composição mobile no portal (`AppShell`).

**Notas de repertório (Ste):** (1) o mecanismo do UI-02/03 chama-se **automatic minimum size** de flex items — flex não encolhe um item abaixo do min-content sem `min-width: 0`; é a causa nº 1 de "ellipsis que não funciona". (2) O toggle sumir atrás do trigger é **paint order**: elementos com `position` pintam acima dos sem `position`, independente de z-index não declarado.
